Amusement Park RDBMS is a database project developed to manage amusement park operations such as customers, rides, ticket booking, staff, and payments. The project uses SQL and relational database concepts like tables, relationships, queries, and CRUD operations for efficient data management.

\# \*\*1\\. Introduction\*\*



Normalization is the systematic process of organizing a relational database to reduce redundancy and ensure data integrity. Each stage, called a normal form, applies a specific rule to the schema. This document formally applies normalization from First Normal Form (1NF) through Third Normal Form (3NF) to every table in the Amusement Park Management System.



Where a table required a change, the change and reason are documented. Where a table already satisfied a normal form, a written justification is provided explaining why - a simple pass statement without explanation is not included.



\# \*\*2\\. Schema Overview\*\*



The schema consists of six tables:



| \*\*Table\*\*           | \*\*Primary Key\*\* | \*\*All Attributes\*\*                                                                              |

| ------------------- | --------------- | ----------------------------------------------------------------------------------------------- |

| \*\*Customers\*\*       | customer\_id     | customer\_id, full\_name, phone                                                                   |

| \*\*Tickets\*\*         | ticket\_id       | ticket\_id, customer\_id, ticket\_type, price, valid\_date                                          |

| \*\*Rides\*\*           | ride\_id         | ride\_id, ride\_name, ride\_type, min\_capacity, max\_capacity, price\_per\_round, status, description |

| \*\*Ride\_Bookings\*\*   | booking\_id      | booking\_id, customer\_id, ride\_id, booked\_at, num\_riders, amount\_paid                            |

| \*\*Ride\_Operations\*\* | operation\_id    | operation\_id, ride\_id, op\_date, total\_rounds, total\_customers                                   |

| \*\*Revenue\*\*         | revenue\_id      | revenue\_id, ride\_id, rev\_date, earnings, source                                                 |



\*\*3\\. First Normal Form (1NF)\*\*



Requirements: every column stores atomic values; each column holds values of a single consistent type; every row is uniquely identified by a primary key; no repeating groups exist.



\## \*\*3.1 Customers\*\*



Satisfies 1NF. Following a real-world workflow optimization, the unnecessary email attribute was permanently removed. The remaining three attributes are strictly atomic: full\_name stores a single identity string per row, and phone holds exactly one contact number. The surrogate primary key customer\_id uniquely identifies each individual customer profile record. There are no repeating groups, composite fields, or multi-valued lists stored in any column.



\## \*\*3.2 Tickets\*\*



Satisfies 1NF. Each row represents a single, independent ticket transaction record. ticket\_id serves as the primary key. ticket\_type is implemented as an atomic ENUM datatype, price is a single scalar DECIMAL field, and valid\_date is a single DATE column indicating when the pass is active for entry. The foreign key customer\_id stores exactly one integer pointer mapping back to the customer profile-not a multi-valued list or array. All column values are atomic and single-typed.



\## \*\*3.3 Rides\*\*



Satisfies 1NF without change. ride\_id uniquely identifies each ride. All attributes are atomic and single-valued. The status column is an ENUM (Active / Maintenance / Closed) ensuring controlled, indivisible values. The description column holds a single text string. min\_capacity and max\_capacity are separate integer columns - they are not combined into one field - which is the correct atomic representation.



\## \*\*3.4 Ride\_Bookings\*\*



Satisfies 1NF without change. booking\_id is the primary key. Each row records one booking event: one customer, one ride, one timestamp, one rider count, one payment amount. num\_riders and amount\_paid are both single numeric values. The two foreign keys (customer\_id and ride\_id) each store a single integer per row - they are not stored as a list even though a customer can book multiple rides, because each booking is its own row. This is the correct relational approach.



\## \*\*3.5 Ride\_Operations\*\*



Satisfies 1NF without change. operation\_id is the primary key. Each row records the daily operational summary for one specific ride on one specific date. total\_rounds and total\_customers are single integer values. A UNIQUE constraint on (ride\_id, op\_date) prevents duplicate summaries for the same ride on the same day, enforcing data integrity without violating 1NF.



\## \*\*3.6 Revenue\*\*



Satisfies 1NF without change. revenue\_id is the primary key. Each row captures one revenue entry: one ride, one date, one earnings figure, one source type. The source column uses an ENUM (Booking / Walk-in / Group / Event). All values are atomic. No column stores a list or composite value.



\## \*\*3.7 Changes Made for 1NF\*\*



The schema was structurally modified to completely exclude the email attribute from the Customers entity to optimize high-throughput walk-in data entry workflows at the ticketing counters. Beyond this optimization, no structural atomicity adjustments were required. The schema satisfies 1NF through single-column surrogate primary keys, separate capacity integers, and strict scalar datatypes across all six tables.



\*\*4\\. Second Normal Form (2NF)\*\*



Requirements: satisfies 1NF, and every non-key attribute is fully functionally dependent on the entire primary key. Partial dependency - where a non-key attribute depends on only part of a composite primary key - violates 2NF. This condition only applies to tables with composite primary keys.



\## \*\*4.1 Composite Key Analysis\*\*



None of the six tables use a composite primary key. Every table uses a single-column surrogate INT AUTO\_INCREMENT primary key. Since partial dependency by definition can only occur when the primary key has two or more columns, and since no table here has a composite primary key, all six tables satisfy 2NF directly.



\## \*\*4.2 Ride\_Operations\*\*



The Ride\_Operations table has a natural composite identifier: the combination of ride\_id and op\_date uniquely identifies one daily summary. However, the design deliberately uses a surrogate operation\_id as the primary key, with a UNIQUE constraint on (ride\_id, op\_date) to enforce the business rule. This means the table has a single-column PK, which eliminates any possibility of partial dependency. This was a deliberate and justified design choice.



\## \*\*4.3 Changes Made for 2NF\*\*



No changes required. The use of surrogate single-column primary keys across all tables eliminated the possibility of partial dependencies by design.



\*\*5\\. Third Normal Form (3NF)\*\*



Requirements: satisfies 2NF, and no non-key attribute is transitively dependent on the primary key through another non-key attribute. In other words, no non-key column should determine another non-key column.



\## \*\*5.1 Customers\*\*



Satisfies 3NF with structural modifications. Originally, ticket\_type, visit\_date, and email were included in this profile table.



\*\*Removal of visit\_date from Customers:\*\* A customer's profile is a permanent entity, whereas visiting the park is a repetitive operational transaction. Storing visit\_date inside Customers means a returning visitor would either overwrite their past tracking data or force the desk operator to duplicate their profile row, violating normalization standards. Shifting this tracking to the Tickets table allows a single customer profile to log unlimited park visits over time.



\*\*Removal of ticket\_type from Customers:\*\* Ticket classification (Adult, Child, Student) is an attribute of the sold financial product, not a fixed personal trait. Storing it in Customers prevents a parent from buying different ticket types for their family under one profile. Moving it to Tickets allows a single transaction to scale dynamically while preserving clean customer demographics.



\*\*Removal of email from Customers:\*\* High visitor throughput at entry gates is a primary amusement park operation. Forcing cashier staff to manually input complex string email addresses for walk-in families creates severe processing bottlenecks and introduces high data-entry error rates. Replacing it with a lean, phone-only requirement ensures fast lookups.



The remaining non-key fields (full\_name, phone) now depend solely, directly, and non-transitively on customer\_id.



\## \*\*5.2 Tickets\*\*



Satisfies 3NF with justification. At first glance, price may appear to depend on ticket\_type (Adult tickets cost 1000, Child 500, etc.), which would create a transitive dependency chain: ticket\_id



\\> ticket\_type > price. However, price is stored explicitly per ticket row to capture the actual amount charged at the exact time of purchase. This allows for seasonal discounts or pricing adjustments over time without altering historical accounting records. Storing it here is a justified, intentional snapshot for data history.



\*\*Renaming Modification (Business Realignment):\*\* The column originally named purchase\_date was explicitly renamed to valid\_date during this version review to resolve the removal of visit\_date from the Customers entity.



\#### \_Why was purchase\_date changed to valid\_date?\_



From an amusement park's business operations stand point, the date a customer buys a ticket is not always the date they visit the park (for example, buying an entry ticket online on Thursday to use in the park on Sunday).



Since visit\_date was purged from the Customers master profile table to fix severe insertion anomalies, the database needed a reliable attribute to track exactly which day a ticket is active. Renaming this field to valid\_date inside the transactional Tickets table serves two critical operational purposes:



\- It allows entry-gate scanners to immediately validate if the ticket is being used on the correct calendar day.

\- It natively replaces the customer profile's old visit\_date attribute, allowing a user to buy tickets for any future dates without causing database duplication or data degradation errors.



\## \*\*5.3 Rides\*\*



Satisfies 3NF. All attributes (ride\_name, ride\_type, min\_capacity, max\_capacity, price\_per\_round, status, description) are direct properties of the ride entity and depend solely on ride\_id. min\_capacity and max\_capacity are both ride-level attributes - one does not determine the other. No transitive dependency exists.



\## \*\*5.4 Ride\_Bookings\*\*



Satisfies 3NF. amount\_paid is computed from num\_riders multiplied by the ride's price\_per\_round. One might argue this creates a transitive dependency. However, price\_per\_round is not stored in Ride\_Bookings - it is stored in the Rides table. Within Ride\_Bookings, amount\_paid depends on booking\_id (i.e., it is a fact about that specific booking event), not on another non-key column within the same table. Storing it preserves the payment amount at the time of booking, independent of any future price changes.



\## \*\*5.5 Ride\_Operations\*\*



Satisfies 3NF. total\_rounds and total\_customers are both direct operational facts about a specific ride on a specific day - they depend on operation\_id. Neither column determines the other. There is no non-key to non-key dependency.



\## \*\*5.6 Revenue\*\*



Satisfies 3NF. rev\_date, earnings, and source all depend directly on revenue\_id. earnings does not depend on source or rev\_date - it is a fact about that specific revenue entry. ride\_id is a foreign key reference, not a non-key determinant.



\## \*\*5.7 Changes Made for 3NF\*\*



Structural changes were executed to achieve strict 3NF compliance and real-world operational logic. The attributes ticket\_type and visit\_date were completely removed from the Customers profile table to eliminate transitive redundancy, moving their logic directly into the Tickets transaction entity where they can scale iteratively over multiple customer visits.



The email field was also purged to streamline customer identity tracking for fast walk-in park workflows. Furthermore, storing price explicitly inside Tickets and amount\_paid within Ride\_Bookings are documented and justified as intentional snapshots needed to preserve historical financial ledger values against future seasonal or base rate modifications.

Each table was checked for redundant columns, repeated data, and overlapping attributes.

\# \*\*1\\. Overview\*\*



This milestone covers three tasks: (1) generating structured synthetic data sufficient to populate all six database tables, (2) documenting the complete dataflow for the Amusement Park Management System, and (3) exporting one CSV file per table for use in Milestone 5 data population.



\# \*\*2\\. Dataset Preparation\*\*



\## \*\*2.1 Approach\*\*



Since the Amusement Park Management System does not use a real-world dataset, structured synthetic data was generated using Python with the Faker library. The data has been systematically cleaned and transformed to ensure complete relational consistency with our \*\*Version 1.3 3NF architecture\*\*.



\## \*\*2.2 Row Counts Per Table\*\*



| \*\*Table\*\*           | \*\*Row Count\*\* | \*\*Notes\*\*                                                                                   |

| ------------------- | ------------- | ------------------------------------------------------------------------------------------- |

| \*\*Rides\*\*           | 10            | Static park asset catalog - seeded once at system setup, never modified by front-end users. |

| \*\*Customers\*\*       | 75            | Master visitor list containing names and standardized Pakistani mobile phone connections.   |

| \*\*Tickets\*\*         | 75            | Point-of-sale gate entry receipts. Stores ticket tiers and calendar access targets.         |

| \*\*Ride\_Bookings\*\*   | 163           | Transactional records capturing customer interactions with specific park rides.             |

\## \*\*2.3 Data Realism \& Consistency Rules\*\*



To meet rigorous data preprocessing requirements, the following validation rules were applied during dataset creation:



\*\*Contact Standardization:\*\* Customer phone numbers match standard domestic mobile patterns (03##########) represented cleanly as text strings. All complex, slow-to-input string attributes (like email) are excluded to optimize gate booth workflows.



\*\*Temporal Tracking Realignment:\*\* The customer profile contains no temporal elements. Instead, entry permissions are tracked using valid\_date within the Tickets table, distributed across an operational window in April and May 2026.



\*\*Tier Assignment Logic:\*\* The attribute ticket\_type is treated as a transactional property of an individual sale within the Tickets table, using an ENUM validation constraint (Adult, Child, Senior, Student).



\*\*Asset Availability Filter:\*\* Ride bookings can only map to rows in the Rides table where status = 'Active'. Under-maintenance assets (such as \_Spinning Cups\_) are excluded from user booking options.



\*\*Capacity Safeguards:\*\* The num\_riders field inside Ride\_Bookings is checked programmatically to ensure it never violates the min\_capacity or max\_capacity constraints defined in the target ride's catalog row.



\*\*Financial Ledger Computations:\*\* The amount\_paid value inside Ride\_Bookings is computed automatically via an equation.



\*\*Revenue Reconciliations:\*\* All Booking-sourced earnings correspond directly to valid records in the Ride\_Bookings junction table. Supplemental rows with Walk-in, Group, and Event origins are added to model non-digital revenue streams accurately.



\## \*\*2.4 Preprocessing Steps Applied\*\*



Before exporting data to clean CSV structures, the following data cleaning procedures were executed:



\- \*\*Redundancy Elimination:\*\* The email and visit\_date attributes were dropped from customers.csv to fulfill 3NF compliance.

\- \*\*Key Uniqueness Verification:\*\* Every primary key consists of a sequentially incremented, non-null integer value to prevent index collisions.

\- \*\*Foreign Key Integrity Audit:\*\* Dependent child columns (customer\_id, ride\_id) were audited against their parent master records to ensure no orphaned references exist.

\- \*\*String Trimming:\*\* Text strings were stripped of leading or trailing whitespaces to maintain clean, predictable data indexing.



\*\*3\\. Dataflow Description\*\*



This section defines how data enters the system, moves through our relational tables, and translates into high-value outputs. This structure reflects the actual updated database configuration.



\## \*\*3.1 Data Entry Points\*\*



| \*\*Entry Point\*\*                | \*\*Target Table\*\* | \*\*Description\*\*                                                                                                                                                                      |

| ------------------------------ | ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |

| \*\*Ride seed script (SQL)\*\*     | Rides            | Run once during setup to populate the catalog with 10 operational rides. This reference data is fixed and cannot be altered by visitors.                                             |

| \*\*Customer registration form\*\* | Customers        | Captured at ticket counters or online forms. Collects stable profile data (full\_name, phone) and auto-assigns a unique customer\_id.                                                  |

| \*\*Ticket pass\*\*                | Tickets          | Populated when an admission pass is purchased. Cashiers assign a customer\_id, select a ticket\_type, and input the scheduled valid\_date. The system automatically provides the price. |

| \*\*Ride booking form\*\*          | Ride\_Bookings    | Created when a customer boards an asset. The system captures customer\_id, ride\_id, and num\_riders. The backend calculates amount\_paid and appends a real-time timestamp.             |

| \*\*Revenue auto-insert\*\*        | Revenue          | Triggered automatically after a transaction occurs in Ride\_Bookings. It records the transaction under the Booking source category.needed.                                            |

| \*\*Nightly\*\* Auditing Utility   | Ride\_Operations  | An automated process aggregates data from Ride\_Bookings at midnight, compiling total daily usage stats (total\_rounds, total\_customers) per ride.                                     |



\## \*\*3.2 Inter-Table Dependencies (Data Movement)\*\*



Data moves through the tables in the following sequence. Each arrow represents a dependency - the child table cannot be populated before its parent.



\\\[Rides\\] ─────────────|─> \\\[Ride\_Bookings\\] ───> \\\[Revenue\\]



| ^



\\\[Customers\\] ──> \\\[Tickets\\] |



| |



|────────────────|



· \*\*Customers -> Tickets:\*\* A ticket cannot be issued without mapping to an established customer record. The column customer\_id inside Tickets acts as a direct foreign key constraint to the master customer ledger.



· \*\*Customers -> Ride\_Bookings <- Rides:\*\* The Ride\_Bookings table resolves the Many-to-Many relationship between visitors and park rides. It cannot ingest data without drawing valid primary key references from both parent tables.



· \*\*Ride\_Bookings -> Revenue:\*\* Upfront transaction amounts captured during ride checkouts pass directly to the financial tracking entity, defining its baseline earnings log.



· \*\*Ride\_Bookings -> Ride\_Operations:\*\* Individual transaction sequences provide the raw data used to generate aggregated end-of-day operations charts.



\## \*\*3.3 Data Output (What Comes Out)\*\*



| \*\*Output\*\*                    | \*\*Description\*\*                                                                                                                                                                                                      |

| ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| \*\*Most popular rides report\*\* | Aggregates total transaction frequencies and headcount distributions from Ride\_Bookings to help management optimize park ride placement and promotions.                                                              |

| \*\*Daily revenue total\*\*       | Compiles all data points within the Revenue table to output daily financial updates for administrative oversight dashboards.                                                                                         |

| \*\*Ride\*\* Efficiency \*\*rate\*\*  | Combines statistics from Ride\_Operations and Rides to evaluate equipment utilization rates:<br><br>Utilization Rate = total\_customers / (total\_rounds × max\_capacity) - shows how full rides are running on average. |

| \*\*Ticket type breakdown\*\*     | Measures ticket allocations across different ticket\_type categories within the Tickets ledger to identify target customer groups.                                                                                    |

| \*\*Customer spending summary\*\* | Combines information across Customers, Tickets, and Ride\_Bookings to track complete financial contributions per visitor, supporting customer loyalty initiatives.                                                    |

| \*\*Revenue by source\*\*         | Audited daily income logs                                                                                                                                                                                            |
